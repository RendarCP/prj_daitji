# 아이템 상세 구조 고도화 및 공유 기능 대응

## 개요

현재 아이템 상세 경험은 `SidePanel` 기반 modal 흐름이 주 진입점이다. `ItemDetailPanelFromData`는 `mode="modal"`과 `mode="page"`를 모두 지원하지만, 실제 주요 탐색 흐름은 `Explorer`, `Dashboard`, `Stackflow`에서 modal 진입을 중심으로 사용하고 있다.

이 구조는 빠른 조회에는 적합하지만, 향후 `공유`, `링크 복사`, `권한 관리`, `외부 링크 진입` 같은 확장 액션을 안정적으로 수용하기에는 한계가 있다. 이 문서는 현재 제약을 정리하고, 장기적으로는 `page-first` 구조를 기준으로 상세 경험을 재정리하면서 공유 기능이 자연스럽게 들어갈 수 있는 방향을 제안한다.

## 현재 구조 요약

### 상세 진입 방식

- `components/features/ItemDetailPanelFromData.tsx`
  - `mode="modal"`과 `mode="page"`를 모두 지원
  - 공통 표시 콘텐츠는 `ItemDetailContent`에 위임
- `components/ui/ItemDetailPanel.tsx`
  - `SidePanel`을 감싼 modal 전용 상세 wrapper
- `app/item/[id]/ItemDetailPageClient.tsx`
  - `ItemDetailPanelFromData`를 `mode="page"`로 사용
- `app/layout.tsx`
  - App Router의 `modal` 슬롯을 통해 modal 레이어를 루트에 공통 렌더링

### 현재 사용 패턴

- 탐색 계열 화면에서는 item tap 시 modal 상세를 먼저 여는 흐름이 중심이다.
- 독립 상세 페이지(`/item/[id]`)도 존재하지만, 현재 구조상 canonical entry라기보다는 별도 상세 진입점에 가깝다.
- `ItemDetailContent`는 콘텐츠 표시 역할에 집중하고 있고, 실제 액션 배치와 wrapper 책임은 modal/page 컨테이너 쪽에 분산되어 있다.

## 현재 한계

### 1. 확장 액션을 놓을 공간이 좁다

현재 상세 화면은 패널 헤더와 하단 CTA에 액션이 분산되어 있다. 지금은 `수정`, `삭제` 수준에서는 운영 가능하지만, 향후 `공유`, `링크 복사`, `즐겨찾기`, `권한 보기`, `참여자 관리` 같은 액션이 추가되면 배치 밀도가 급격히 올라간다.

### 2. 공유 링크 진입과 modal UX가 충돌한다

공유 기능이 생기면 사용자는 링크를 통해 특정 아이템 상세에 직접 진입하게 된다. 이 경우 사용자가 기대하는 맥락은 독립 페이지 기반 상세 화면인데, quick-view 성격의 modal은 외부 진입의 기준 화면으로 적합하지 않다.

### 3. URL 중심 상태 관리가 약하다

modal 중심 구조에서는 “현재 어떤 아이템을 보고 있는지”를 브라우저 URL, 히스토리, 새로고침, 외부 링크 진입과 자연스럽게 연결하기 어렵다. 공유 기능은 결국 URL 기반 진입을 전제로 하므로, canonical detail surface가 필요하다.

### 4. quick view와 canonical detail의 목적이 다르다

탐색 중 빠르게 보는 화면과, 링크로 진입해 액션까지 수행하는 상세 화면은 목적이 다르다. 현재는 두 성격을 하나의 구조로 같이 감당하고 있어, 향후 고도화 시 설계 충돌이 생길 가능성이 높다.

## 목표 구조

### 핵심 원칙

- 장기 기준 상세 경험은 `Canonical Detail Page`로 정리한다.
- modal은 `Quick View Modal` 역할로 축소한다.
- 공유 및 확장 액션의 기준 surface는 `page`로 둔다.
- 외부 공유 링크의 canonical entry는 `/item/[id]`로 통일한다.

### 역할 정의

#### Quick View Modal

- 목록/탐색 중 빠르게 아이템 정보를 훑어보는 용도
- 최소한의 조회와 빠른 이동 중심
- 중요 액션은 제한적으로 제공
- 필요 시 `전체 페이지로 보기` CTA 제공

#### Canonical Detail Page

- 아이템 상세의 기준 화면
- 공유, 링크 복사, 수정, 삭제, 권한 관련 액션의 기본 surface
- 브라우저 URL, 새로고침, 외부 진입, 뒤로가기 맥락을 자연스럽게 수용

#### Share Entry / Shared Link Entry

- 공유 링크로 직접 진입하는 엔트리
- 항상 page 기반으로 해석
- 접근 가능 여부, 권한 없음, 존재하지 않음 등의 상태를 명확히 표현

## 권장 UX 흐름

### 탐색 화면에서 아이템을 열 때

- 기본 흐름은 현행처럼 quick view modal을 유지할 수 있다.
- 다만 modal은 빠른 확인 중심으로 사용한다.
- 중요 액션은 page에서만 노출하거나, modal에서는 축약된 형태로만 제공한다.
- modal 내부에는 `전체 페이지로 보기` CTA를 둘 수 있다.

### 공유 링크로 아이템을 열 때

- 항상 `/item/[id]` page를 기본 엔트리로 사용한다.
- 공유 링크 진입 시 modal을 띄우는 방식은 사용하지 않는다.
- 사용자는 독립 페이지 문맥에서 콘텐츠와 액션을 확인해야 한다.

### 상세 페이지에서 공유 액션을 사용할 때

- page 헤더 액션 영역에 `공유`, `링크 복사`, 향후 `권한 관리`를 배치할 수 있도록 설계한다.
- 실제 공유 설정 UI는 modal 또는 bottom sheet로 열 수 있지만, 호출 주체는 page가 되어야 한다.
- 공유 가능 상태가 아닐 경우 액션을 비활성화하고 이유를 안내한다.

## 구현 가이드

### 구조 원칙

- `ItemDetailContent`는 표시 전용 콘텐츠로 유지한다.
- 액션 배치, 화면 레이아웃, page/modal 분기 책임은 wrapper 계층에서 담당한다.
- `ItemDetailPanelFromData`는 공통 데이터 조립 역할을 유지하되, 액션 정책은 `mode`에 따라 분기할 수 있어야 한다.

### 액션 구조 정리 방향

- `headerActions`는 modal/page에서 서로 다른 구성을 허용한다.
- 필요 시 `ItemDetailActions` 같은 별도 액션 컴포넌트로 분리해, `mode`별 허용 액션을 명시적으로 관리한다.
- 예시 정책:
  - `modal`: 닫기, 수정, 전체 페이지로 보기
  - `page`: 공유, 링크 복사, 수정, 삭제, 추가 확장 액션

### page-first 이행 방향

1. `/item/[id]`를 상세 경험의 기준 엔트리로 재정의한다.
2. modal은 quick view 목적에 맞게 정보와 액션을 축약한다.
3. 공유 기능은 page 기준으로 먼저 설계하고, modal에는 필요 최소 연결만 둔다.

## 공유 기능 대응 요구사항

### 최소 요구사항

- 공유 링크 생성 또는 링크 복사
- 공유 가능/불가능 상태 구분
- 권한 없음 처리
- 존재하지 않는 아이템 처리
- 외부 링크 진입 시 안정적인 상세 페이지 렌더링

### 상태별 대응

#### 공유 가능 상태

- page에서 `공유` 액션 노출
- 링크 복사 또는 공유 설정 진입 허용

#### 공유 불가능 상태

- 액션 비활성화
- 비활성 이유 안내

#### 접근 실패 상태

- 권한 없음
- 삭제되었거나 존재하지 않는 아이템
- 만료되었거나 유효하지 않은 공유 링크

이 상태들은 모두 page 기준에서 명확한 메시지와 fallback action을 제공해야 한다.

### 후속 상세 명세 범위

아래 항목은 별도 문서로 분리 가능하다.

- 인증/권한 모델
- 공유 토큰 정책
- 초대 기반 협업 모델
- 서버 API 및 데이터 스키마

이번 문서에서는 원칙 수준만 다룬다.

## 시나리오 예시

### 1. 탐색 중 빠르게 확인하는 경우

사용자는 탐색 화면에서 아이템을 탭하고 `Quick View Modal`을 연다. 여기서는 요약 정보와 핵심 액션만 확인하고, 추가 작업이 필요하면 `전체 페이지로 보기`로 이동한다.

### 2. 링크로 직접 상세를 여는 경우

사용자는 공유 링크를 통해 특정 아이템을 연다. 이 경우 항상 `Canonical Detail Page`로 진입하며, 브라우저 URL과 화면 맥락이 일치해야 한다.

### 3. 상세 페이지에서 공유 버튼을 누르는 경우

사용자는 page 헤더 액션에서 `공유`를 누르고 링크 복사 또는 공유 설정 UI로 진입한다. 공유 액션은 page 기준에서 제공된다.

### 4. 공유 링크로 들어왔지만 권한이 없는 경우

사용자는 상세 내용을 보기 전에 권한 없음 상태를 확인한다. 이 경우 modal이 아니라 독립 페이지에서 명확한 오류 메시지와 복귀 동선을 제공해야 한다.

## 단계별 고도화 로드맵

### 1단계. 상세 page 기준 experience 정리

- `/item/[id]`를 상세의 기준 화면으로 재정의
- page에서 확장 가능한 header action 영역 확보

### 2단계. modal을 quick view로 역할 축소

- modal 노출 정보와 액션 범위 축소
- `전체 페이지로 보기` 동선 추가 검토

### 3단계. 공유 액션 및 링크 진입 설계

- page 기준 공유 액션 추가
- 공유 링크 진입 시나리오 정리

### 4단계. 권한 및 협업 확장

- 공유 범위, 권한 모델, 협업 기능 고도화
- 필요 시 별도 API/도메인 문서로 확장

## 문서 적용 시 확인 포인트

- 현재 코드 구조와 실제 진입 흐름이 문서 내용과 일치해야 한다.
- 왜 modal만으로는 공유 기능이 불편한지 구체적인 UX 제약이 드러나야 한다.
- 향후 구현자가 이 문서만 보고 `page-first + modal quick-view` 방향을 이해할 수 있어야 한다.
